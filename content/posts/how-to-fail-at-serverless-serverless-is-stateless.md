<p id="">This blog series is an extended version of a talk I gave at <a href="https://nyc.serverlessdays.io/" id="">Serverless Days NYC 2022</a>. The goal is to share ways that a "friend of mine" has failed at serverless to help level up the community. Just as the transportation industry shares accident report analysis with the whole industry to improve safety, we in the software community need to do the same. This is my attempt to do that.<br></p><p id=""><br></p><p id="">I'm sure there is some product X that has feature Y that fixes issue Z. I either didn't know about it at the time, didn't have the budget, couldn't get it through vendor/security review, product wouldn‚Äôt schedule it with our engineers or it didn't comply with enterprise architecture's plan for our software roadmap. As some guy once said: ‚ÄúYou go to production with the tools you have, not the tools you want.‚Äù</p><p id=""><br></p><p id=""><strong id="">Note:</strong>‚Äç</p><ul id=""><li id="">Names have been changed to protect the innocent and not so innocent.</li><li id="">Only egos (and wallets) were harmed by the failures recounted here.</li></ul><p id=""><br></p><h1 id=""><strong id="">Serverless is Stateless</strong></h1><p id=""><br></p><p id="">All the architecture posts talk about serverless as stateless computing. How serverless is so scalable because it scales out to satisfy increased load, and scales in when not needed. This all makes sense. Lambda is like a big thread pool in the sky. Got it.</p><p id=""><br></p><p id="">I can set some parameters to manage the maximum and minimum number of threads and AWS would handle all the hard work of hosting, spinning up, spinning down and networking the instances together. Awesome-sauce.</p><p id="">‚Äç</p><figure id="" class="w-richtext-figure-type-image w-richtext-align-center" data-rt-type="image" data-rt-align="center"><div id=""><img src="https://uploads-ssl.webflow.com/60d0f077b69e2d8f2d246168/62c86713fed3b513be122ce8_i-dont-remember.gif" loading="lazy" id="" width="auto" height="auto"></div><figcaption id="">lambda be like<br></figcaption></figure><p id=""><br></p><p id="">Well, almost...</p><p id=""><br></p><h3 id=""><strong id="">Use Case</strong></h3><p id=""><br></p><p id="">Background job that receives batches (10's-100's) of PDFs from the client, splits them apart into individual pages (1-1000 per PDF) and does analysis on each page. The arrival rate from the client is lumpy. Primarily US working hours on a 5 min interval. Response is measured in hours. This feels like a perfect HelloWorld use case for StepFunctions doing orchestration and Lambda doing image manipulation.</p><p id=""><br></p><h4 id="">Lambda logic:</h4><ul id=""><li id="">Invoke with S3 path to file</li><li id="">Read 10+mb files from S3 to /tmp</li><li id="">Process file</li><li id="">Write results to S3</li><li id="">Respond with S3 path to transformed file</li></ul><p id="">‚Äç</p><p id="">Serverless FTW!</p><p id=""><br></p><p id="">Everything works great until... inconsistent out of disk space errors, or no space left on device (depending on your OS/language stack).</p><p id=""><br></p><h2 id="">How to Fail</h2><p id=""><br></p><h3 id=""><strong id="">Problem</strong></h3><p id=""><br></p><p id="">Periodically a step function run will fail due to a Lambda invoke failing on no space left on device (or similar depending on os and language). Running the same file in dev/test doesn't reproduce the error. Re-running the step function in production succeeds.</p><p id="">‚Äç</p><figure id="" class="w-richtext-figure-type-image w-richtext-align-center" data-rt-type="image" data-rt-align="center"><div id=""><img src="https://uploads-ssl.webflow.com/60d0f077b69e2d8f2d246168/62c866a26c2399e457733a1f_lambda-pac-man.png" loading="lazy" id="" width="auto" height="auto"></div><figcaption id="">lambda pac man loves eating data and leaving it in /tmp</figcaption></figure><p id=""><br></p><p id=""><br></p><h3 id=""><strong id="">Solution</strong></h3><p id=""><br></p><p id="">Well... lambda instances have a writable `/tmp` folder where you can store temporary files. And somewhere in the AWS docs they describe how instances of your Lambda function are reused between invocations.</p><p id=""><br></p><p id="">Some quick googling gets you a solution of clearing `/tmp` at the start and/or end of each invocation (<a href="https://stackoverflow.com/questions/44108712/aws-lambda-release-tmp-storage-after-each-execution">stack overflow ftw</a>). Till that fix makes it to production you can just re-execute any failed step function runs.</p><p id=""><br></p><p id="">Nothing quite like shipping code copy-pasted from Stack Overflow. Nice.</p><p id=""><br></p><p id="">Success! Party!<br></p><figure id="" class="w-richtext-figure-type-image w-richtext-align-center" data-rt-type="image" data-rt-align="center"><div id=""><img src="https://uploads-ssl.webflow.com/60d0f077b69e2d8f2d246168/62c867cdc310db14bbac6405_leo-congrats.gif" loading="lazy" id="" width="auto" height="auto"></div><figcaption id="">Stack Overflow code FTW!</figcaption></figure><p id=""><br></p><p id="">‚Äç</p><p id="">until... wait what? out of memory errors? WTF!</p><p id=""><br></p><h3 id=""><strong id="">Problem v2</strong></h3><p id=""><br></p><p id="">Periodically a step function run fails. The Lambda invoke error is no memory left (or similar depending on os and language). Running the same file in dev/test doesn't reproduce the error. Re-running the step function in production succeeds. Turning on debug or releasing new builds adding additional logging suppress the error rate. WTF (why the failure?) Load tests don't reproduce the error.</p><p id=""><br></p><p id="">The explanation that AWS just fails on 1 in ~100k lambda invokes starts to wear thin with your Ops team that is getting paged to click a button. But we could just automate the re-run!</p><p id=""><br></p><p id="">Deep inside accepting it "just fails" violates some axiom about how computers works. This is knowable and damn-it I shall know. So you strap the lambda down setting max concurrency to 1. Turn up the load test. Let‚Äôs do this.</p><p id=""><br></p><p id="">At 12k documents the lambda invoke fails for out of memory. Interesting.</p><p id=""><br></p><p id="">Rerun the load test. At 12k documents the lambda invoke fails for out of memory.</p><p id=""><br></p><p id="">If an error is repeatable it isn't random. 12k is an important number. It also happens to be the maximum number of file descriptors allowed for the linux OS the lambda is running on.</p><p id=""><br></p><p id="">Back to google. Apparently the image manipulation library I am using has a history of bugs with leaving open file descriptors. Looks like each invoke to lambda is leaving another file descriptor open in the OS file descriptor table.</p><p id=""><br></p><p id="">The lambda will then fail when it hits 12k unless it is recycled. The lambda instance can be recycled due to:</p><ul id=""><li id="">a code deployment: new code, new instances</li><li id="">update to environment variable: hence why turning on debugging suppressed errors.</li><li id="">scaling down: a burst in load will scale out new instances and then scale back down. Only some instances survive.</li></ul><p id=""><br></p><h3 id=""><strong id="">Solution v2</strong></h3><p id=""><br></p><p id="">Expand the IAM role permissions to allow the lambda permission to update itself (your security team might give you side eye for this, fair warning, send them this blog post, I'm sure it will convince them).</p><p id=""><br></p><p id="">After completing the work of an invoke, call the lambda api to set an environment variable to a random value. We dubbed this the "self-immolation" solution.</p><p id=""><br></p><figure id="" class="w-richtext-figure-type-image w-richtext-align-center" data-rt-type="image" data-rt-align="center"><div id=""><img src="https://uploads-ssl.webflow.com/60d0f077b69e2d8f2d246168/62c868256c23997e757361fa_lambda-self-immolation.png" loading="lazy" id="" width="auto" height="auto"></div><figcaption id="">This lambda will self destruct in 3... 2... 1...</figcaption></figure><p id=""><br></p><p id="">Hacky?
